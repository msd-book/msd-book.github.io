Proxy pattern — a deeper dive

Short summary: a Proxy is an object that implements the same interface as some real subject and controls access to it — adding lazy-loading, caching, access control, logging, retries, throttling, security, or remote communication — without changing clients. Think of it as a small, focused adapter that sits between callers and the real implementation.

Why use a Proxy?

  • Lazy initialization / Virtual Proxy: delay costly object creation until needed (large images, DB connections).
  • Caching: avoid repeated expensive calls (remote RPCs, heavy computations).
  • Protection / Access control: enforce permissions before delegating.
  • Remote proxy / Stub: hide network details so client uses local interface.
  • Logging/Tracing/Instrumentation: add metrics without changing core logic.
  • Resilience: add retries, backoff, circuit breakers.
  • Testing: provide controllable stand-ins for integration points.

Apply when you need cross-cutting behaviour that should not pollute the real subject nor client code.

Where not to use it

  • When the real subject is cheap to create and access.
  • When behaviour would be better expressed via middleware/aspect (e.g., cross-cutting frameworks), DI pipelines, or decorator chains with clear separation of concerns.
  • When a proxy would introduce surprising semantics for clients (state differences, identity issues).

Types of proxies (common)

  • Virtual Proxy: lazy-create the heavy object.
  • Caching Proxy: store results to avoid recomputation/remote calls.
  • Protection Proxy: check permissions.
  • Remote Proxy / Stub: marshal/unmarshal over network.
  • Logging / Monitoring Proxy: capture metrics/traces.
  • Smart Reference: manage lifecycle (count refs, manage pooling).

Concrete C# example — Before / After

We'll use a simple scenario: an IDocumentService that fetches and renders documents from a remote store. Initially clients call a concrete DocumentService directly; later we introduce a DocumentServiceProxy that adds caching, access control, and lazy fetch.

This is a ready-to-run .NET console example. Save as Program.cs and run with dotnet run in a .NET SDK environment.

Before — no proxy (clients call the concrete service directly)

using System;
using System.Threading.Tasks;

namespace ProxyBefore
{
    public class Document
    {
        public string Id { get; }
        public string Content { get; }
        public Document(string id, string content) { Id = id; Content = content; }
    }

    // Concrete service used directly by callers
    public class DocumentService
    {
        public async Task<Document> FetchDocumentAsync(string id)
        {
            // Simulate a remote call
            await Task.Delay(1000);
            return new Document(id, $"Content for {id} at {DateTime.UtcNow}");
        }

        public void Render(Document doc)
        {
            Console.WriteLine($"[Render] Document {doc.Id}: {doc.Content}");
        }
    }

    class Program
    {
        static async Task Main()
        {
            var svc = new DocumentService();

            // Client A
            var d1 = await svc.FetchDocumentAsync("doc-1");
            svc.Render(d1);

            // Client B re-fetches same doc (another expensive call)
            var d2 = await svc.FetchDocumentAsync("doc-1");
            svc.Render(d2);
        }
    }
}

Problems: every fetch is expensive; no access control; adding logging/metrics requires editing DocumentService or every client.

After — Proxy introduced (interface + real service + proxy)

using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

namespace ProxyAfter
{
    // Domain model
    public class Document
    {
        public string Id { get; }
        public string Content { get; }
        public Document(string id, string content) { Id = id; Content = content; }
    }

    // Common interface
    public interface IDocumentService
    {
        Task<Document> FetchDocumentAsync(string id, string callerId);
        void Render(Document doc);
    }

    // The "real" service (expensive remote)
    public class RemoteDocumentService : IDocumentService
    {
        public async Task<Document> FetchDocumentAsync(string id, string callerId)
        {
            // Simulate remote delay and fetch
            await Task.Delay(1000);
            return new Document(id, $"[Remote] Content for {id} at {DateTime.UtcNow}");
        }

        public void Render(Document doc)
        {
            Console.WriteLine($"[Render] Document {doc.Id}: {doc.Content}");
        }
    }

    // The Proxy: caching + protection + logging + lazy
    public class DocumentServiceProxy : IDocumentService
    {
        private readonly IDocumentService _real;
        private readonly ConcurrentDictionary<string, Document> _cache = new();
        private readonly Func<string, bool> _authorizationCheck;

        public DocumentServiceProxy(IDocumentService real, Func<string, bool> authorizationCheck)
        {
            _real = real ?? throw new ArgumentNullException(nameof(real));
            _authorizationCheck = authorizationCheck ?? (_ => false);
        }

        public async Task<Document> FetchDocumentAsync(string id, string callerId)
        {
            if (!_authorizationCheck(callerId))
                throw new UnauthorizedAccessException($"Caller '{callerId}' not authorized to fetch documents.");

            // Fast path: cached
            if (_cache.TryGetValue(id, out var cached))
            {
                Console.WriteLine($"[Proxy] Cache HIT for {id}");
                return cached;
            }

            Console.WriteLine($"[Proxy] Cache MISS for {id}. Fetching from remote...");
            var doc = await _real.FetchDocumentAsync(id, callerId);

            // Store into cache (thread-safe via ConcurrentDictionary)
            _cache.TryAdd(id, doc);

            return doc;
        }

        public void Render(Document doc)
        {
            // Add logging around render calls
            Console.WriteLine($"[Proxy] Rendering {doc.Id} (length={doc.Content?.Length ?? 0})");
            _real.Render(doc);
        }

        // Optional: ability to invalidate cache (explicit)
        public void Invalidate(string id) => _cache.TryRemove(id, out _);
    }

    class Program
    {
        static async Task Main()
        {
            IDocumentService real = new RemoteDocumentService();

            // simple auth: only "alice" allowed
            Func<string, bool> auth = caller => caller == "alice";

            var proxy = new DocumentServiceProxy(real, auth);

            try
            {
                // Caller alice (authorized) -> cache miss -> remote fetch
                var d1 = await proxy.FetchDocumentAsync("doc-1", "alice");
                proxy.Render(d1);

                // Another request by alice -> cache hit (no remote call)
                var d2 = await proxy.FetchDocumentAsync("doc-1", "alice");
                proxy.Render(d2);

                // Caller bob (unauthorized) -> throws
                var d3 = await proxy.FetchDocumentAsync("doc-2", "bob");
                proxy.Render(d3);
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Exception: {ex.Message}");
            }
        }
    }
}

What proxy added

  • Authorization check before delegating.
  • Caching of previously fetched documents to avoid repeated remote calls.
  • Logging around cache hits/misses and rendering.
  • No client changes: clients still depend on IDocumentService.

Notes on this refactor

  • Introducing an interface (IDocumentService) enables both the real implementation and proxy to be used interchangeably.
  • Proxy encapsulates cross-cutting concerns; real service focuses on core work (fetch/render).
  • Cache implemented with ConcurrentDictionary for thread-safety in this example; in production you'd consider eviction, max-size, TTL, memory pressure.
  • Authorization is provided by an injected Func<string, bool> — in real apps you'd inject a proper auth service.

Lint-style checklist for code reviews

Use these as short, reviewer-facing checklist items:

  1. Interface present: Is there a small, clear interface the proxy and real subject both implement? (e.g., IDocumentService)
  2. Single responsibility: Is the proxy only adding cross-cutting behavior (caching/auth/logging) and not mixing domain logic?
  3. Transparent behavior: Does the proxy preserve the subject’s semantics (exceptions, side-effects)? Ensure it doesn't silently drop errors.
  4. Thread-safety: Are caches and mutable state in the proxy thread-safe and properly synchronized?
  5. Cache correctness: Is there a clear invalidation/TTL policy and consideration for stale data?
  6. Fail-fast & errors: Does the proxy fail in a way clients expect (not swallow exceptions unless explicitly intended)?
  7. Performance measurement: Has any additional latency or blocking behaviour been measured (proxy shouldn't become a bottleneck)?
  8. No identity leaks: The proxy should not expose references allowing clients to bypass it and call the real subject directly.
  9. Test coverage: Unit tests for cache hit/miss, auth success/failure, and error propagation must exist.
  10. Composition clarity: If multiple decorators/proxies exist, ordering is explicit and documented (e.g., auth before cache?).

Common pitfalls and how to avoid them

  1. Thread-safety bugs in caching

    • Pitfall: using non-thread-safe collections or check-then-act patterns leading to duplicate remote calls or race conditions.
    • Avoid: use ConcurrentDictionary, Lazy<T> patterns, or proper locks. Consider double-checked locking carefully.

  2. Stale cache / invalidation complexity

    • Pitfall: returning stale data because no TTL or invalidation.
    • Avoid: document cache semantics, add TTL/ETag-based validation, provide explicit invalidation hooks.

  3. Leaking real subject / bypass

    • Pitfall: clients obtain a direct reference to the real subject and bypass proxy.
    • Avoid: control DI/creation so only proxy is injected; do not expose the real subject through API.

  4. Changing semantics (surprising behaviour)

    • Pitfall: proxy swallows exceptions or changes return types subtly.
    • Avoid: preserve contract semantics; if you add retries, surface retry policy via configuration and logs.

  5. Proxy becomes large & complex

    • Pitfall: adding too many responsibilities (auth, caching, metrics, retry, validation) makes the proxy hard to maintain.
    • Avoid: split concerns: chain small decorators/proxies, or use middleware/aspect frameworks when appropriate.

  6. Proxy performance bottleneck

    • Pitfall: proxy adds synchronization or blocking causing latency at high concurrency.
    • Avoid: design non-blocking caches, use asynchronous patterns correctly, benchmark.

  7. Testing difficulties

    • Pitfall: hard to unit-test due to tight coupling or global state.
    • Avoid: dependency-inject collaborators, make policy pluggable (inject cache, auth, logging).

  8. Identity equality and reference issues

    • Pitfall: clients rely on object.ReferenceEquals semantics or cast to concrete type; proxies break these expectations.
    • Avoid: document proxy usage; prefer interface-based clients.

  9. Security assumptions

    • Pitfall: relying on proxy for security while other code may bypass it.
    • Avoid: enforce security at boundaries (API gateway, DB, RBAC) not only via in-process proxy.

Quick patterns & recipes

  • Virtual proxy (lazy): create a proxy that holds a Lazy<T> to the heavy subject.
  • Caching proxy: wrap fetch calls; store in memory or distributed cache; use cache key hygiene.
  • Protection proxy: validate identity/roles before delegating; fail-fast with UnauthorizedAccessException.
  • Remote proxy (stub): serialize requests, handle timeouts/retries, translate remote exceptions to domain ones.
  • Instrumentation proxy: record durations and counts, integrate with metrics system.

When to prefer other approaches

  • If you only need logging/metrics for HTTP/API calls, consider middleware or AOP.
  • If you need flexible pipelines (auth → validation → caching → endpoint), consider a pipeline/middleware design rather than a monolithic proxy.
  • For distributed caches, use existing caching infra (Redis) rather than in-process dictionaries unless you need locality.

Final advice (practical)

  • Start small: implement the minimal proxy behaviour you need (e.g., caching), keep it well-tested.
  • Keep the proxy thin — extract complex policy into collaborators you can test independently (cache provider, auth provider, retry policy).
  • Document the proxy’s guarantees (cache TTL, auth policy, retry semantics) so clients and reviewers understand the tradeoffs.
