Interface Segregation Principle (ISP) — deeper dive

Short definition: ISP says clients should not be forced to depend on interfaces they do not use. In practice that means prefer many small, focused interfaces to one large “fat” interface.

Why it matters in modern software

  • Reduces unnecessary coupling — changes to a large interface ripple to many clients.
  • Makes implementations simpler — classes only implement what they actually need.
  • Improves testability — you can mock a small interface easily.
  • Enables clearer abstractions in microservices, libraries, and event-driven code where consumers vary widely.

When to apply ISP

  • Public APIs or libraries consumed by different clients.
  • Hardware/adapter layers (drivers, printers, payment gateways).
  • Large services where different consumers require different subsets of behavior.
  • Any place a single interface grows beyond a single cohesive responsibility.

When not to over-apply

  • Don’t create a million tiny interfaces for the sake of it — keep cohesion. If two methods are always used together by clients, they can reasonably live in the same interface.

Concrete example (Java)

We’ll start with a common anti-pattern — a fat OfficeMachine interface that forces implementers to provide methods they don’t need.

Before — ISP violation

// Fat interface
public interface OfficeMachine {
    void print(Document d);
    void scan(Document d);
    void fax(Document d);
    void sendEmail(Document d);
}

// A simple printer forced to implement unused members
public class SimplePrinter implements OfficeMachine {
    @Override
    public void print(Document d) { /* printing logic */ }

    @Override
    public void scan(Document d) {
        throw new UnsupportedOperationException("scan not supported");
    }

    @Override
    public void fax(Document d) {
        throw new UnsupportedOperationException("fax not supported");
    }

    @Override
    public void sendEmail(Document d) {
        throw new UnsupportedOperationException("email not supported");
    }
}

// A client depends on OfficeMachine but only needs print
public class PrintService {
    private final OfficeMachine machine;

    public PrintService(OfficeMachine machine) {
        this.machine = machine;
    }

    public void printDoc(Document d) {
        machine.print(d);
    }
}

Problems here:

  • SimplePrinter is forced to carry stub/unimplemented methods.
  • PrintService depends on a larger interface than necessary.
  • Changes to OfficeMachine affect unrelated implementers.

After — ISP-aligned refactor

Split into focused interfaces and compose them where needed.

// Segregated interfaces
public interface Printer {
    void print(Document d);
}

public interface Scanner {
    void scan(Document d);
}

public interface Fax {
    void fax(Document d);
}

public interface EmailSender {
    void sendEmail(Document d);
}

// Compose interfaces when a device supports multiple capabilities
public interface MultiFunctionDevice extends Printer, Scanner {}

// Implementations implement only what they need
public class SimplePrinter implements Printer {
    @Override
    public void print(Document d) { /* printing logic */ }
}

public class AllInOneDevice implements MultiFunctionDevice, Fax, EmailSender {
    @Override
    public void print(Document d) { /* print */ }

    @Override
    public void scan(Document d) { /* scan */ }

    @Override
    public void fax(Document d) { /* fax */ }

    @Override
    public void sendEmail(Document d) { /* send email */ }
}

// Clients depend on the smallest interface they need
public class PrintService {
    private final Printer printer;

    public PrintService(Printer printer) {
        this.printer = printer;
    }

    public void printDoc(Document d) {
        printer.print(d);
    }
}

Benefits after refactor:

  • SimplePrinter is minimal and has no unsupported methods.
  • PrintService explicitly expresses its dependency on Printer only.
  • New behaviors (e.g., sendFax) can be added to small interfaces without touching unrelated classes.

Practical migration steps

  1. Identify fat interfaces — grep for interfaces with many unrelated methods or throw new UnsupportedOperationException.
  2. Group methods by cohesive responsibility — create new focused interfaces (e.g., Printer, Scanner).
  3. Refactor implementations — implement the small interfaces they actually need.
  4. Update clients — make clients depend on the narrow interfaces rather than the fat one.
  5. Provide adapter/composite types — if some code expects the old interface, create an adapter that composes the smaller interfaces (temporary shim).
  6. Run tests and update any DI/registrations.
  7. Deprecate old fat interfaces gradually, not suddenly.

Example adapter for backward compatibility

If you must keep the old interface temporarily:

public class OfficeMachineAdapter implements OfficeMachine {
    private final Optional<Printer> printer;
    private final Optional<Scanner> scanner;
    private final Optional<Fax> fax;
    private final Optional<EmailSender> emailSender;

    public OfficeMachineAdapter(Printer p, Scanner s, Fax f, EmailSender e) {
        this.printer = Optional.ofNullable(p);
        this.scanner = Optional.ofNullable(s);
        this.fax = Optional.ofNullable(f);
        this.emailSender = Optional.ofNullable(e);
    }

    @Override
    public void print(Document d) {
        printer.orElseThrow(() -> new UnsupportedOperationException("print not supported")).print(d);
    }

    @Override
    public void scan(Document d) {
        scanner.orElseThrow(() -> new UnsupportedOperationException("scan not supported")).scan(d);
    }

    // ... fax and sendEmail similar
}

This lets you migrate consumers gradually.

Additional real-world patterns where ISP helps

  • Microservices: expose multiple small endpoints rather than a single all-in-one RPC that many clients partially use.
  • SDKs/APIs: split large client interfaces into capability-focused clients (e.g., StorageClient, AuthClient).
  • Plugin systems: plugins implement only the lifecycle hooks they need.
  • Event handlers: specific event interfaces instead of a single catch-all handler.

Tradeoffs & gotchas

  • Too many tiny interfaces can increase complexity. Aim for logical cohesion: group methods used together.
  • Over-segregation can complicate DI bindings; plan for composite adapters or marker interfaces where needed.
  • Consider evolution: version interfaces thoughtfully and prefer adding new specific interfaces over changing existing ones.

Lint-style checklist for code reviews (ISP-focused)

Use these as short checklist items in PR templates:

  1. Interface cohesion: Does the interface represent a single responsibility? ✔ / ✖
  2. Client usage: Are any implementing classes forced to provide methods they never use or throw UnsupportedOperationException? ✔ / ✖
  3. Minimal dependency: Do consumers depend on the narrowest interface needed? (Prefer Printer over OfficeMachine.) ✔ / ✖
  4. Method grouping: Are methods grouped by behavior commonly used together? (If not, split.) ✔ / ✖
  5. Avoid god interfaces: Reject interfaces that mix unrelated concerns (I/O, network, business logic). ✔ / ✖
  6. Migration adapters: If changing a public interface, is there an adapter/shim for backward compatibility? ✔ / ✖
  7. Tests exist: Are there unit tests for each small interface contract and for composite adapters? ✔ / ✖
  8. DI registrations: Have DI/container registrations been updated to bind to the smaller interfaces? ✔ / ✖
  9. Documentation: Are interfaces documented with intended clients and responsibilities? ✔ / ✖
  10. Practical cohesion check: If >2 disparate clients would need different subsets of methods, consider splitting the interface. ✔ / ✖

Quick checklist for implementing ISP in a module

  • Start by listing all methods and their consumers.
  • For each consumer, create an interface that contains only the methods it needs.
  • Keep interfaces named for the capability (Printer, Authenticator, MetricsEmitter).
  • Use composition for devices/objects that support multiple capabilities.
  • Add unit tests that mock only the focused interfaces.
  • Review public APIs for backwards compatibility and provide adapters if needed.
