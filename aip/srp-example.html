Monolithic example (anti-SRP)

// src/order-service/OrderService.ts
import db from "../infra/db";
import Stripe from "stripe";
import EmailClient from "../infra/email";
import Metrics from "../infra/metrics";

const stripe = new Stripe(process.env.STRIPE_KEY!, { apiVersion: "2020-08-27" });

export default class OrderService {
  async placeOrder(userId: string, items: Array<{id:string,qty:number}>, cardToken: string) {
    // 1. validate input
    if (!items || items.length === 0) throw new Error("empty cart");

    // 2. calculate total (coupon/tax/shipping)
    const total = items.reduce((s, it) => s + /*lookup price*/ 10 * it.qty, 0);

    // 3. charge card
    const charge = await stripe.charges.create({ amount: total * 100, currency: "usd", source: cardToken });

    // 4. persist order
    const order = await db.orders.insert({ userId, items, total, chargeId: charge.id });

    // 5. send confirmation email
    await EmailClient.send(userId, `order ${order.id} placed`);

    // 6. record metrics
    Metrics.increment("orders.created");

    // 7. write audit log
    await db.audit.insert({ type: "order_created", orderId: order.id, ts: Date.now() });

    return order;
  }

  async listOrders(userId: string) {
    // also mixes presentation & persistence concerns
    const rows = await db.orders.find({ userId });
    return rows.map(r => ({ id: r.id, total: r.total, items: r.items }));
  }
}

What's wrong (reasons-to-change)

This single class mixes:

  • Input validation and business rules (cart validation, pricing)
  • Payment integration (Stripe)
  • Persistence (db.orders)
  • Notification/emailing (EmailClient)
  • Telemetry/metrics (Metrics)
  • Auditing
  • Presentation mapping (listOrders mapping) Different teams, changes, or incidents could affect each: finance/APIs, DB schema, email templates, or metrics/observability.

SRP-focused refactor — goals

  • Each reason-to-change becomes a separate collaborator with a clear contract.
  • OrderService becomes an orchestrator of pure responsibilities: validate → pay → persist → notify → audit → metrics.
  • Improve testability (unit tests for business logic without network calls).
  • Keep one place to read the high-level flow.

Rewritten orchestrator (single responsibility: orchestrate order placement)

// src/order-service/OrderCreator.ts
import { OrderRepository, PaymentGateway, Notifier, Auditor, Metrics, PriceService, CartItem, OrderRecord } from "./ports";

export class OrderCreator {
  constructor(
    private priceService: PriceService,
    private paymentGateway: PaymentGateway,
    private repo: OrderRepository,
    private notifier: Notifier,
    private auditor: Auditor,
    private metrics: Metrics
  ) {}

  // responsibility: orchestrate creating an order (business flow)
  async placeOrder(userId: string, items: CartItem[], paymentToken: string): Promise<OrderRecord> {
    // validate
    if (!items || items.length === 0) throw new Error("empty cart");

    // compute business total (delegated)
    const total = await this.priceService.computeTotal(items);

    // charge payment (delegated)
    const payment = await this.paymentGateway.charge(Math.round(total * 100), "usd", paymentToken);

    // persist (delegated)
    const order: OrderRecord = { userId, items, total, paymentId: payment.id, createdAt: Date.now() };
    const saved = await this.repo.save(order);

    // notify (delegated)
    await this.notifier.sendOrderConfirmation(userId, saved.id!);

    // audit & metrics (delegated)
    await this.auditor.record("order_created", { orderId: saved.id, userId: userId });
    this.metrics.increment("orders.created");

    return saved;
  }
}

OrderCreator has one responsibility: coordinate the order creation flow. All other concerns are explicit collaborators.

Testing strategy after refactor

  • Unit tests for OrderCreator: mock PriceService, PaymentGateway, OrderRepository, Notifier, Auditor, Metrics. Assert orchestration rules:

    • Calls priceService.computeTotal once with correct items
    • Calls paymentGateway.charge with expected cents
    • Persists with correct payload
    • Notifier and Auditor are invoked
    • Metrics increment called

  • Unit tests for PriceService: pure rules for discounts/taxes.

  • Integration tests for infra adapters:

    • SqlOrderRepo against test DB (or SQLite in-memory)
    • StripeGateway mocked or a VCR-like replay; do not hit real Stripe in CI

  • Contract tests: ensure adapter shapes conform to ports.

  • End-to-end tests: full flow with service stubs for external systems when necessary.

Migration / incremental refactor plan (practical)

  1. Add interfaces (ports) — non-breaking.
  2. Create thin adapter classes that delegate to the existing code (initially they call back into the monolith).
  3. Implement new orchestrator (OrderCreator) but have HTTP controllers still call old OrderService until tests pass.
  4. Wire orchestrator into a new endpoint or feature flag and run A/B or shadow mode to observe behaviour.
  5. Switch the HTTP handler to the orchestrator, keep adapters reusing old infra during transition.
  6. Refactor old class incrementally: remove code moved to adapters.
  7. Remove duplicated logic and delete the old monolith class when no callers remain.

This incremental approach lets you keep the system running while moving responsibilities out.

Operational considerations (microservices / distributed)

  • If you extract a responsibility into another service (e.g., Payments service):

    • Use clear contracts (API/gRPC), version them.
    • Consider eventual consistency and sagas for multi-step flows (payment succeeded but persistence failed).
    • Add retries/backoff and idempotency keys for external calls.
    • Monitor end-to-end traces (distributed tracing) so you can see the flow across services — helps identify where to split next.
    • Be pragmatic: co-locate responsibilities when latency or transactionality matters; split when teams/SLA/ownership require it.

Trade-offs & guidance

  • SRP reduces cognitive load per component — but increases number of components. Keep discoverability: use clear names and good directory structure.
  • Don’t extract too early: prefer extraction when you need to change something, or tests become painful.
  • Favor composition and small, well-named ports. Over-abstraction (80 interfaces for 20 lines of code) is also wasteful.

Mini checklist to use in code reviews

  • What are the reasons this file would change? List them in the PR description.
  • Does the class/module expose one clear concept? (Yes/No)
  • Are cross-cutting concerns separated (logging/metrics/auth)?
  • Are external systems behind an interface/port?
  • Is business logic testable without infra? (Unit tests without network/DB)
  • If there are many commits to unrelated areas in the same file, extract.
