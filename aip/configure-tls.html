Following is a practical, copy-pasteable guide to get TLS running on NGINX and a full checklist of verification steps you can run immediately.

1) Quick outline

  1. Get a certificate (Let’s Encrypt / commercial / self-signed).
  2. Place cert + private key on the server.
  3. Configure NGINX server block(s) for HTTPS (listen 443 ssl, certificate paths, TLS options).
  4. Test the config and reload NGINX.
  5. Verify with nginx -t, curl, openssl s_client, and an external scanner (e.g., SSL Labs or testssl.sh). The NGINX docs and Mozilla’s config recommendations are good references while doing this. (Nginx)

2) Obtain a certificate

Recommended (Let’s Encrypt + Certbot — automated, free): (on Ubuntu/Debian)

sudo apt update
sudo apt install certbot python3-certbot-nginx
# get and auto-configure (interactive)
sudo certbot --nginx -d example.com -d www.example.com

Certbot can automatically add the ssl_certificate/ssl_certificate_key lines to your NGINX site. (There are equivalent instructions for other distros; DigitalOcean has a good tutorial.) (DigitalOcean)

Manual / commercial / testing (self-signed):

# self-signed (testing only)
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/example.key \
  -out /etc/ssl/certs/example.crt \
  -subj "/CN=example.com"

Place cert and key where NGINX can read them (e.g. /etc/letsencrypt/live/example.com/fullchain.pem and /etc/letsencrypt/live/example.com/privkey.pem for Certbot).

3) Example NGINX HTTPS server block (minimal + recommended extras)

Replace example.com and paths with your values.

server {
    listen 80;
    server_name example.com www.example.com;
    # Redirect all HTTP to HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com www.example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Recommended protocol lines (modern + fallback): TLS 1.2 and 1.3
    ssl_protocols TLSv1.2 TLSv1.3;

    # Prefer server ciphers and use a modern cipher list — use Mozilla's generator for exact string
    ssl_prefer_server_ciphers on;
    ssl_ciphers "ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:\
                 ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384";

    # Session and performance
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1h;

    # (Optional) OCSP stapling (requires ssl_trusted_certificate pointing to CA chain)
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/example.com/chain.pem;

    # HTTP security headers
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    add_header X-Frame-Options SAMEORIGIN;
    add_header X-Content-Type-Options nosniff;

    # Your usual root / proxy_pass / index etc
    root /var/www/example;
    index index.html index.htm;
    location / { try_files $uri $uri/ =404; }
}

Notes

  • Use Mozilla SSL Config Generator to create the precise ssl_ciphers, ssl_protocols, and additional directives matching your OpenSSL/nginx versions and compatibility needs. It's a trusted source for secure configs. (ssl-config.mozilla.org)
  • Generate DH params only if you need them (e.g. openssl dhparam -out /etc/ssl/certs/dhparam.pem 2048) and reference in config if desired.
  • OCSP stapling requires the server to be able to fetch OCSP responses from the CA (network access) and the ssl_trusted_certificate to be set to the CA chain. See NGINX docs for details. (Nginx)

4) Test NGINX and reload

# syntax check
sudo nginx -t

# if OK, reload gracefully
sudo systemctl reload nginx

If nginx -t fails, read the error and fix the config before reloading. (NGINX docs cover the directives and examples.) (Nginx)

5) Verification checklist — commands & what to look for

5.1 Quick HTTP -> HTTPS check (browser)

Browse to https://example.com and confirm padlock. Browsers show certificate chain and expiry when you click the lock.

5.2 curl — basic TLS info

curl -I -sS https://example.com/ -o /dev/null -w "%{url_effective} %{ssl_verify_result} %{ssl_cipher} %{ssl_protocol}\n"

%{ssl_verify_result} should be 0 (success). The cipher and protocol show what was negotiated.

5.3 openssl s_client — inspect cert and negotiated protocol

openssl s_client -connect example.com:443 -servername example.com -showcerts

Look for:

  • Verify return code: 0 (ok) — certificate chain validates locally.
  • Protocol : TLSv1.3 (or TLSv1.2) — negotiated protocol.
  • Certificate details and chain printed above. Use -cipher to test specific ciphers. The -servername flag sends SNI for name-based virtual hosts. (OpenSSL s_client is the canonical CLI tester.) (Oracle Docs)

5.4 check OCSP stapling (if enabled)

From openssl s_client output, when stapling works you’ll see OCSP Response: content and verify return:... entries. If not, check NGINX error log and ensure the server can reach CA OCSP servers and ssl_trusted_certificate is set. (NGINX docs include stapling notes.) (Nginx)

5.5 Test with an external scanner (deep checks)

  • Qualys SSL Labs — paste your public hostname into SSL Labs Server Test for a full report (protocols, ciphers, chain, HSTS, certificate, vulnerabilities). This is the most common external verification. (SSL Labs)
  • testssl.sh or sslyze — run from your shell for scripted testing and CI. testssl.sh examines supported protocols, weak ciphers, heartbleed, etc. Many teams run this in CI. (MOSS)

Example local run (download testssl.sh and run):

./testssl.sh --fast example.com

6) Extra hardening (optional, follow Mozilla/NGINX guidance)

  • Disable TLS 1.0 / 1.1 (use TLS 1.2 & 1.3 only).
  • Prefer ECDHE ciphers; avoid RC4, DES, 3DES.
  • Enable HSTS (Strict-Transport-Security) once confident HTTP->HTTPS redirect is stable.
  • Enable OCSP stapling.
  • Use strong certs (RSA 2048+ or ECDSA).
  • Keep OpenSSL and NGINX up to date. Mozilla’s Server Side TLS documentation and the NGINX admin guide provide practical recommended settings. (Mozilla Wiki)

7) Common troubleshooting

  • Verify return code: 21 or similar from openssl s_client — usually missing intermediate CA; point ssl_trusted_certificate (or provide full chain as ssl_certificate depending on CA).
  • Browser says mixed content (insecure HTTP resources) — fix references to load over HTTPS.
  • OCSP stapling errors — ensure CA OCSP endpoint reachable and ssl_trusted_certificate contains the chain.
  • Old clients can't connect — you may need to accept older ciphers or TLS versions for compatibility (tradeoff: security vs legacy clients). See Mozilla generator to pick a compatibility tier. (ssl-config.mozilla.org)
