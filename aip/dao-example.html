What is a DAO (Data Access Object)?

A Data Access Object (DAO) is a design pattern that encapsulates all access to a data source (database, file, remote service) behind a simple API. The DAO exposes CRUD and query operations for domain objects and hides the low-level persistence details (SQL, connections, ORM specifics).

Short: DAO = domain-centric interface + concrete implementation that talks to the DB.

Why use a DAO?

  • Separation of concerns — keeps persistence code out of business logic.
  • Testability — you can mock or fake the DAO in unit tests.
  • Single place for DB rules — central location for SQL, transactions, retry logic, logging.
  • Swapability — swap DB implementations (e.g. ADO.NET → Dapper → EF Core) without touching business code.
  • Consistency & safety — standardize parameterization, connection handling, transactions.

Where & how to use it

  • Use DAOs at the boundary between domain/services and the persistence layer.
  • Keep DAOs focused on persistence operations only; no business rules.
  • Prefer small, focused DAO interfaces (per aggregate/entity) rather than one huge database service.
  • Implement async I/O, parameterized queries, proper connection/transaction management, and expose a testable interface.

3) Connection factory (abstraction for testability)

using System.Data;

public interface IDbConnectionFactory
{
    Task<IDbConnection> CreateOpenConnectionAsync(CancellationToken ct = default);
}

public class SqlConnectionFactory : IDbConnectionFactory
{
    private readonly string _connectionString;
    public SqlConnectionFactory(string connectionString) => _connectionString = connectionString;

    public async Task<IDbConnection> CreateOpenConnectionAsync(CancellationToken ct = default)
    {
        var conn = new Microsoft.Data.SqlClient.SqlConnection(_connectionString);
        await conn.OpenAsync(ct).ConfigureAwait(false);
        return conn;
    }
}

5) ADO.NET implementation (SqlCustomerDao)

using System;
using System.Collections.Generic;
using System.Data;
using System.Threading;
using System.Threading.Tasks;

public class SqlCustomerDao : ICustomerDao
{
    private readonly IDbConnectionFactory _connectionFactory;
    public SqlCustomerDao(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    public async Task<int> AddAsync(Customer customer, CancellationToken ct = default)
    {
        const string sql = @"
INSERT INTO Customers (FirstName, LastName, Email)
VALUES (@FirstName, @LastName, @Email);
SELECT CAST(SCOPE_IDENTITY() AS int);
";
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        AddParameter(cmd, "@FirstName", customer.FirstName);
        AddParameter(cmd, "@LastName", customer.LastName);
        AddParameter(cmd, "@Email", customer.Email);

        var result = await ExecuteScalarAsync(cmd, ct).ConfigureAwait(false);
        return Convert.ToInt32(result);
    }

    public async Task<bool> UpdateAsync(Customer customer, CancellationToken ct = default)
    {
        const string sql = @"
UPDATE Customers
SET FirstName = @FirstName, LastName = @LastName, Email = @Email
WHERE Id = @Id;
";
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        AddParameter(cmd, "@Id", customer.Id);
        AddParameter(cmd, "@FirstName", customer.FirstName);
        AddParameter(cmd, "@LastName", customer.LastName);
        AddParameter(cmd, "@Email", customer.Email);

        var affected = await ExecuteNonQueryAsync(cmd, ct).ConfigureAwait(false);
        return affected > 0;
    }

    public async Task<bool> DeleteAsync(int id, CancellationToken ct = default)
    {
        const string sql = "DELETE FROM Customers WHERE Id = @Id;";
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        AddParameter(cmd, "@Id", id);
        var affected = await ExecuteNonQueryAsync(cmd, ct).ConfigureAwait(false);
        return affected > 0;
    }

    public async Task<Customer?> GetByIdAsync(int id, CancellationToken ct = default)
    {
        const string sql = "SELECT Id, FirstName, LastName, Email, CreatedUtc FROM Customers WHERE Id = @Id;";
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        AddParameter(cmd, "@Id", id);
        await using var reader = await ExecuteReaderAsync(cmd, ct).ConfigureAwait(false);
        if (!await ReadAsync(reader, ct).ConfigureAwait(false)) return null;
        return MapCustomer(reader);
    }

    public async Task<Customer?> GetByEmailAsync(string email, CancellationToken ct = default)
    {
        const string sql = "SELECT Id, FirstName, LastName, Email, CreatedUtc FROM Customers WHERE Email = @Email;";
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        AddParameter(cmd, "@Email", email);
        await using var reader = await ExecuteReaderAsync(cmd, ct).ConfigureAwait(false);
        if (!await ReadAsync(reader, ct).ConfigureAwait(false)) return null;
        return MapCustomer(reader);
    }

    public async Task<IReadOnlyList<Customer>> GetAllAsync(CancellationToken ct = default)
    {
        const string sql = "SELECT Id, FirstName, LastName, Email, CreatedUtc FROM Customers ORDER BY Id;";
        var list = new List<Customer>();
        await using var conn = await _connectionFactory.CreateOpenConnectionAsync(ct).ConfigureAwait(false);
        await using var cmd = conn.CreateCommand();
        cmd.CommandText = sql;
        await using var reader = await ExecuteReaderAsync(cmd, ct).ConfigureAwait(false);
        while (await ReadAsync(reader, ct).ConfigureAwait(false))
            list.Add(MapCustomer(reader));
        return list;
    }

    #region helpers (ADO abstractions to keep code short)
    // The following helpers use dynamic typing on IDbCommand / IDataReader
    // to keep the example compact. In production you may prefer typed provider types or Dapper.

    private static void AddParameter(IDbCommand cmd, string name, object? value)
    {
        var p = cmd.CreateParameter();
        p.ParameterName = name;
        p.Value = value ?? DBNull.Value;
        cmd.Parameters.Add(p);
    }

    private static Task<object?> ExecuteScalarAsync(IDbCommand cmd, CancellationToken ct)
    {
        if (cmd is Microsoft.Data.SqlClient.SqlCommand sqlCmd)
            return sqlCmd.ExecuteScalarAsync(ct).ContinueWith(t => (object?)t.Result, ct);
        // fallback synchronous wrapper:
        return Task.FromResult(cmd.ExecuteScalar());
    }

    private static Task<int> ExecuteNonQueryAsync(IDbCommand cmd, CancellationToken ct)
    {
        if (cmd is Microsoft.Data.SqlClient.SqlCommand sqlCmd)
            return sqlCmd.ExecuteNonQueryAsync(ct);
        return Task.FromResult(cmd.ExecuteNonQuery());
    }

    private static Task<IDataReader> ExecuteReaderAsync(IDbCommand cmd, CancellationToken ct)
    {
        if (cmd is Microsoft.Data.SqlClient.SqlCommand sqlCmd)
            return sqlCmd.ExecuteReaderAsync(ct).ContinueWith(t => (IDataReader)t.Result, ct);
        return Task.FromResult(cmd.ExecuteReader());
    }

    private static Task<bool> ReadAsync(IDataReader reader, CancellationToken ct)
    {
        if (reader is Microsoft.Data.SqlClient.SqlDataReader sqlReader)
            return sqlReader.ReadAsync(ct);
        return Task.FromResult(reader.Read());
    }

    private static Customer MapCustomer(IDataReader r)
    {
        return new Customer
        {
            Id = r.GetInt32(r.GetOrdinal("Id")),
            FirstName = r.GetString(r.GetOrdinal("FirstName")),
            LastName = r.GetString(r.GetOrdinal("LastName")),
            Email = r.GetString(r.GetOrdinal("Email")),
            CreatedUtc = r.GetDateTime(r.GetOrdinal("CreatedUtc"))
        };
    }
    #endregion
}

Notes on the implementation

  • Parameterized queries to avoid SQL injection.
  • IDbConnectionFactory keeps connection creation centralized and testable.
  • Async APIs everywhere for scalable I/O.
  • The helper methods provide a small shim to use IDbCommand/IDataReader async methods only when the provider supports them; replace with provider-specific APIs in production or use Dapper/EF Core to reduce boilerplate.

6) Fake DAO for unit tests

public class FakeCustomerDao : ICustomerDao
{
    private readonly List<Customer> _store = new();
    private int _nextId = 1;

    public Task<int> AddAsync(Customer customer, CancellationToken ct = default)
    {
        customer.Id = _nextId++;
        customer.CreatedUtc = DateTime.UtcNow;
        _store.Add(customer);
        return Task.FromResult(customer.Id);
    }

    public Task<bool> UpdateAsync(Customer customer, CancellationToken ct = default)
    {
        var idx = _store.FindIndex(c => c.Id == customer.Id);
        if (idx < 0) return Task.FromResult(false);
        _store[idx] = customer;
        return Task.FromResult(true);
    }

    public Task<bool> DeleteAsync(int id, CancellationToken ct = default)
    {
        var removed = _store.RemoveAll(c => c.Id == id) > 0;
        return Task.FromResult(removed);
    }

    public Task<Customer?> GetByIdAsync(int id, CancellationToken ct = default)
        => Task.FromResult(_store.Find(c => c.Id == id));

    public Task<Customer?> GetByEmailAsync(string email, CancellationToken ct = default)
        => Task.FromResult(_store.Find(c => c.Email.Equals(email, StringComparison.OrdinalIgnoreCase)));

    public Task<IReadOnlyList<Customer>> GetAllAsync(CancellationToken ct = default)
        => Task.FromResult((IReadOnlyList<Customer>)_store.ToArray());
}

Use FakeCustomerDao to unit-test service/business logic without a database.

8) Example usage in a service

public class CustomerService
{
    private readonly ICustomerDao _dao;
    public CustomerService(ICustomerDao dao) => _dao = dao;

    public async Task<Customer> RegisterAsync(string firstName, string lastName, string email, CancellationToken ct = default)
    {
        if (string.IsNullOrWhiteSpace(email)) throw new ArgumentException("email");
        var existing = await _dao.GetByEmailAsync(email, ct).ConfigureAwait(false);
        if (existing != null) throw new InvalidOperationException("Email already used.");

        var c = new Customer { FirstName = firstName, LastName = lastName, Email = email };
        c.Id = await _dao.AddAsync(c, ct).ConfigureAwait(false);
        return c;
    }
}

Because CustomerService only depends on ICustomerDao, you can inject FakeCustomerDao in tests.

Additional best practices & pitfalls

Best practices

  • Expose domain-friendly methods (e.g., FindActiveByEmailAsync) instead of raw SQL callers.
  • Keep transactions at a higher level (service layer) when multiple DAOs are involved; expose overloads that accept an IDbTransaction if needed.
  • Log queries and execution time for slow query monitoring.
  • Use parameterized queries (always).
  • Consider an ORM (EF Core/Dapper) if it reduces boilerplate; DAO pattern still applies (DAO wraps ORM).

Common pitfalls

  • Placing business rules in DAOs.
  • Returning raw SQL rows / DataTable — prefer domain objects.
  • Not handling connection/transaction lifecycle properly (leaks).
  • Tight coupling to a specific DB provider in business code (avoid by using interfaces).

TL;DR

A DAO isolates persistence behind a small interface so your business code is testable, swap-able, and simpler. Implement it as an interface + concrete class (or wrapper around your ORM), do async I/O, parameterize queries, centralize connection creation, and provide fakes for tests. Is the DAO pattern still relevant with modern frameworks like ASP.NET Core and ORMs?

Yes — DAO is still relevant, but its shape changes depending on how you use your ORM and what architectural style you follow. The pattern hasn’t disappeared; it has simply evolved.

Below is a practical, modern view that applies to real ASP.NET Core systems.

Why DAO still matters

Even with EF Core, Dapper, or other ORMs, you still have the same core problem: You need a boundary between your domain logic and the details of how data is persisted.

Modern systems benefit from DAOs because:

1) ORMs don’t eliminate the need for separation of concerns

Business logic should not be calling DbContext directly everywhere. If you do that, persistence decisions leak into the entire codebase.

DAO (or Repository) gives you a stable façade:

  • You control what queries exist.
  • You provide domain-friendly operations (e.g., GetActiveCustomers()) instead of raw LINQ queries scattered everywhere.
  • You avoid coupling domain code to EF Core’s API.

2) Testability still requires abstraction

If services depend on DbContext, unit tests require in-memory EF Core or a test DB. If services depend on ICustomerDao (or ICustomerRepository), unit tests run with simple in-memory fakes/mocks.

Fakes are much simpler than EF Core mocks.

3) You gain freedom to evolve persistence

Your schema and ORM choice will change over time. DAO isolates that churn.

It lets you:

  • move from EF to Dapper for performance-heavy queries
  • combine multiple data sources behind a single domain operation
  • introduce caching layers transparently

4) DAO is cheap to implement

A small interface per aggregate root and a class wrapping ORM queries is minimal work and adds real clarity.

When DAO is very useful in modern ASP.NET Core systems

  • medium/large systems with multiple teams
  • systems that need clean layers (API → domain → persistence)
  • systems that may change persistence strategies (e.g., SQL + Redis + search engine)
  • systems with complex domain logic where queries must be curated
  • high testability requirements

When DAO may feel unnecessary

DAO might be overkill if:

  • the entire application is a small CRUD web app
  • domain logic is extremely thin
  • the persistence mechanism is unlikely to change
  • you’re comfortable with business logic living directly in EF Core LINQ

In those cases, using DbContext directly is fine.

How DAO evolves with modern ORMs

In older days, DAOs were very SQL-heavy. In modern practice:

  • With EF Core, DAO often becomes a thin wrapper:

    public class CustomerDao : ICustomerDao
{
    private readonly AppDbContext _db;

    public Task<Customer?> GetByEmailAsync(string email)
        => _db.Customers.SingleOrDefaultAsync(c => c.Email == email);

    public Task AddAsync(Customer c)
        => _db.Customers.AddAsync(c).AsTask();
}

    It hides EF details and ensures services do not perform ad-hoc queries.

  • With Dapper, DAO becomes a place to centralize SQL.

  • With microservices, DAO is often called a Repository, but the role is the same.

The pattern survives because the underlying need — a stable persistence boundary — survives.

Practical guidance (what experienced teams do)

Keep DAOs small

1–3 domain operations per DAO is typical.

Avoid over-abstracting the ORM

You don’t need a generic IRepository<T> — it hides your intent. Use explicit DAO interfaces per entity/aggregate.

Let services own business logic

DAOs should not validate rules, check invariants, or calculate state transitions.

Prefer returning domain models

Avoid leaking ORM-specific entities.

Short answer

Yes, DAO is still relevant. Frameworks and ORMs automate persistence, but they don’t remove the need for a clean boundary, stable domain API, improved testability, and architectural flexibility. The pattern remains a valuable tool—simpler in some apps, more important in others.
