Strategy pattern — why, where, how (C# deep dive)

Short version: Strategy extracts interchangeable algorithms into small classes behind a common interface so you can switch behavior at runtime, test algorithms independently, and avoid conditionals and duplicated code.

When to use it

  • You have a class with multiple if/else or switch blocks selecting algorithms (pricing, validation, compression, serialization, payment, routing).
  • Algorithms vary independently from the clients that use them.
  • You need to swap behavior at runtime (A/B test, feature flags) or inject different implementations for testing.
  • You want to make algorithms independently testable and open for extension.

Trade-offs

    • Clear separation of concerns, easier testing, single-responsibility.
    • Easier to add new strategies without touching existing code.
  • – More classes to manage (but usually worth it).
  • – Slight runtime indirection (negligible in almost all business apps).

Real example: payment processing (C# console app)

We’ll start with a before (monolithic) implementation, then show an after using the Strategy pattern (two variants: simple injected strategy, and a strategy registry for runtime selection). All code is ready-to-run in a .NET console app.

BEFORE — monolithic class with conditionals

Program.cs (before)

using System;

namespace StrategyBefore
{
    enum PaymentType { CreditCard, PayPal, Crypto }

    class PaymentRequest
    {
        public PaymentType Type { get; }
        public decimal Amount { get; }
        public string Payload { get; } // e.g. token, email, wallet

        public PaymentRequest(PaymentType type, decimal amount, string payload)
        {
            Type = type;
            Amount = amount;
            Payload = payload;
        }
    }

    class PaymentProcessor
    {
        // Monolithic method with branching logic
        public bool Process(PaymentRequest req)
        {
            switch (req.Type)
            {
                case PaymentType.CreditCard:
                    // pretend we call external gateway
                    Console.WriteLine($"Charging {req.Amount:C} to card token {req.Payload}");
                    // ... lots of card-specific logic (validation, 3DS, retries) ...
                    return true;

                case PaymentType.PayPal:
                    Console.WriteLine($"Creating PayPal charge for {req.Amount:C} account {req.Payload}");
                    // ... PayPal-specific flows ...
                    return true;

                case PaymentType.Crypto:
                    Console.WriteLine($"Sending {req.Amount} crypto to wallet {req.Payload}");
                    // ... crypto-specific flows ...
                    return true;

                default:
                    throw new NotSupportedException($"Payment type {req.Type} not supported");
            }
        }
    }

    class Program
    {
        static void Main()
        {
            var processor = new PaymentProcessor();

            var r1 = new PaymentRequest(PaymentType.CreditCard, 49.99m, "card_tok_123");
            var r2 = new PaymentRequest(PaymentType.PayPal, 19.95m, "alice@example.com");
            var r3 = new PaymentRequest(PaymentType.Crypto, 0.02m, "0xWalletAddr");

            processor.Process(r1);
            processor.Process(r2);
            processor.Process(r3);
        }
    }
}

Problems with this:

  • PaymentProcessor mixes all algorithms — violates Single Responsibility.
  • Hard to unit test individual payment flows.
  • Adding a new payment type requires editing Process (risk of regressions).
  • Conditionals proliferate.

PaymentService.cs (context)

namespace StrategyAfter
{
    // Context holds a strategy and uses it. Strategy can be swapped at runtime.
    public class PaymentService
    {
        private IPaymentStrategy _strategy;

        public PaymentService(IPaymentStrategy strategy)
        {
            _strategy = strategy ?? throw new ArgumentNullException(nameof(strategy));
        }

        public void SetStrategy(IPaymentStrategy strategy) => _strategy = strategy ?? throw new ArgumentNullException(nameof(strategy));

        public bool Process(PaymentRequest request) => _strategy.Process(request);
    }
}

Program.cs (usage)

using System;

namespace StrategyAfter
{
    class Program
    {
        static void Main()
        {
            var cardStrategy = new CreditCardStrategy();
            var paypalStrategy = new PayPalStrategy();
            var cryptoStrategy = new CryptoStrategy();

            var svc = new PaymentService(cardStrategy);
            svc.Process(new PaymentRequest(49.99m, "card_tok_123"));

            svc.SetStrategy(paypalStrategy);
            svc.Process(new PaymentRequest(19.95m, "alice@example.com"));

            svc.SetStrategy(cryptoStrategy);
            svc.Process(new PaymentRequest(0.02m, "0xWalletAddr"));
        }
    }
}

Notes (Variant A):

  • Good when client knows strategy or receives it via DI.
  • Simple and very testable: unit test each *Strategy independently.
  • To choose strategies by runtime data, you need a factory/registry (Variant B).

Variant B — Strategy registry/factory for runtime selection

Often you need to pick a strategy based on request metadata (like PaymentType). Use a registry/factory that maps keys to strategies — still Strategy pattern, but adds a selection mechanism.

IStrategyFactory.cs

public interface IStrategyFactory<TKey, TStrategy>
{
    TStrategy Get(TKey key);
}

PaymentStrategyFactory.cs

using System;
using System.Collections.Generic;

public class PaymentStrategyFactory : IStrategyFactory<PaymentType, IPaymentStrategy>
{
    private readonly IReadOnlyDictionary<PaymentType, IPaymentStrategy> _map;
    public PaymentStrategyFactory(IDictionary<PaymentType, IPaymentStrategy> map)
    {
        _map = new Dictionary<PaymentType, IPaymentStrategy>(map);
    }

    public IPaymentStrategy Get(PaymentType key)
    {
        if (!_map.TryGetValue(key, out var s)) throw new KeyNotFoundException($"No strategy for {key}");
        return s;
    }
}

PaymentServiceWithFactory.cs

public class PaymentServiceWithFactory
{
    private readonly IStrategyFactory<PaymentType, IPaymentStrategy> _factory;

    public PaymentServiceWithFactory(IStrategyFactory<PaymentType, IPaymentStrategy> factory)
    {
        _factory = factory;
    }

    public bool Process(PaymentType type, PaymentRequest request)
    {
        var strategy = _factory.Get(type);
        return strategy.Process(request);
    }
}

Program.cs (usage with registry)

using System;
using System.Collections.Generic;

class Program
{
    static void Main()
    {
        var map = new Dictionary<PaymentType, IPaymentStrategy>
        {
            { PaymentType.CreditCard, new CreditCardStrategy() },
            { PaymentType.PayPal, new PayPalStrategy() },
            { PaymentType.Crypto, new CryptoStrategy() }
        };

        var factory = new PaymentStrategyFactory(map);
        var svc = new PaymentServiceWithFactory(factory);

        svc.Process(PaymentType.CreditCard, new PaymentRequest(49.99m, "card_tok_abc"));
        svc.Process(PaymentType.PayPal, new PaymentRequest(19.95m, "alice@example.com"));
        svc.Process(PaymentType.Crypto, new PaymentRequest(0.02m, "0xWallet"));
    }
}

Notes (Variant B):

  • Fits apps where you need runtime selection by key.
  • Integrates well with DI containers — register all IPaymentStrategy implementations and build the map automatically.

How this improves the codebase (practical benefits)

  • Testability: Each strategy is a small unit to test in isolation (mock network calls via dependency injection inside strategy).
  • Extensibility: Add a new payment method by adding a new IPaymentStrategy implementation and registering it — no change to PaymentService.
  • Single Responsibility: PaymentService becomes orchestration; strategies encapsulate algorithm details.
  • Runtime flexibility: Swap strategies dynamically (feature flags, A/B tests, tenant-specific behavior).
  • Observability: Instrument strategies separately for metrics/tracing.

Lint-style checklist for code reviews (strategy-specific)

Use this checklist when reviewing code that uses Strategy pattern or when proposing to refactor to it:

  1. Single Responsibility

    • Does the class do one thing? (Strategy implementations should contain algorithm-specific logic only.)

  2. Interface clarity

    • Is the strategy interface small and focused? (No fat interfaces; prefer one method or a cohesive set of methods.)

  3. No type checks

    • Are there if/else or switch statements determining behavior by type inside the context? (If yes, prefer registry/factory with strategy mapping.)

  4. Dependency Injection

    • Are strategies injected (constructor injection) and registered with the DI container? Avoid new inside service code when DI is appropriate.

  5. Runtime selection

    • If selecting strategies at runtime, is there a clean factory/registry? Avoid ad-hoc selection logic scattered across code.

  6. Immutability / Thread-safety

    • Are strategy instances stateless or otherwise thread-safe? If not, ensure proper scoping (transient vs singleton) is used.

  7. Error handling

    • Does the factory/service handle missing strategies gracefully (clear exception or fallback)? Avoid silent failures.

  8. Small method surfaces

    • Are strategy methods small and focused? Large methods should be split.

  9. Logging & Metrics

    • Are strategies instrumented for error reporting and metrics separately (so you can see which algorithm is failing or slow)?

  10. Tests

    • Unit tests for each strategy covering success + failure paths.
    • Integration tests for the registry/factory and the context using a sample of strategies.

  11. Naming

    • Strategy implementations should have clear names: CreditCardStrategy, PayPalStrategy, CryptoStrategy (not Strategy1, Strategy2).

  12. Open for extension, closed for modification

    • Adding a new strategy should not require changing existing strategies or the context (verify in PR).

  13. Configuration

    • If strategies are toggleable by config/feature flag, ensure the selection logic reads config from a single place and is covered by tests.

  14. Performance

    • If strategies are expensive to construct, check DI scope and caching; avoid constructing heavy resources repeatedly.

Quick recommendations for real projects

  • Keep strategies small and injectable — if a strategy needs external clients (HTTP, DB), inject those clients into the strategy.
  • Use a registry or DI naming pattern for runtime selection (e.g., IEnumerable<IPaymentStrategy> and a small initializer builds the map).
  • Prefer stateless strategies so you can register them as singletons safely.
  • Put shared helper code (common validations) in a separate service or base class to avoid duplication without forcing strategies to share state.
