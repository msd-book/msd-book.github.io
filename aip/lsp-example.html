Deep dive: Liskov Substitution Principle (LSP) — why, where, and how

Short version: LSP says a subtype must be usable wherever its supertype is expected without surprising the caller. Practically, that means subclasses must preserve the behavioral contract (preconditions, postconditions, invariants) of their parent types.

If that fails, polymorphism breaks: code that works with the base type will behave incorrectly with the subtype.

Why LSP matters (practical consequences)

  • Predictability: Callers rely on the contract of the base type. Violations cause subtle bugs.
  • Testability: Tests for the base type should pass for any subtype. If they don't, your tests are lying.
  • Extensibility: Proper subtyping lets you add new implementations without changing client code.
  • API stability: Public base types define expectations for downstream users — breaking them is an API bug.

Where LSP commonly fails in real systems

  • Mutable state and overridden setters (classic Rectangle/Square).
  • Strengthening preconditions in subclasses (e.g., subclass requires extra non-nullness).
  • Weakening postconditions (returns less than promised).
  • Changing exception behavior (throwing new unchecked exceptions the base didn't promise).
  • Violating invariants (e.g., allowed ranges change).
  • Using inheritance for code reuse rather than subtype/semantic compatibility.

In modern systems, watch for LSP issues in:

  • SDKs / public libraries
  • Microservice contracts (if a service returns different shapes for polymorphic types)
  • Domain models (ORM entities, value objects)
  • UI component hierarchies (assumptions about lifecycle/props)
  • Concurrent or actor-based code where substitutability implies thread-safety guarantees

Exact rules to keep in mind (contract-style)

  1. Signatures: Method signatures must remain compatible (types, arity).
  2. Preconditions cannot be strengthened in a subtype. If base accepts x, subtype must accept at least all those x.
  3. Postconditions cannot be weakened. If base guarantees y, subtype must guarantee y.
  4. Invariants of supertype must hold for subtype objects.
  5. No surprising side effects — resource/visibility/exception semantics should remain compatible.
  6. History constraint: Subtype must not allow changes that break the external history expected by clients (e.g., mutating previously immutable contract).

Real class/module example — classic Rectangle / Square

We'll use Java for clarity (static types show intent), but the idea applies everywhere.

Before — LSP violation

// Base type
public class Rectangle {
    protected int width;
    protected int height;

    public Rectangle(int w, int h) {
        this.width = w;
        this.height = h;
    }

    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }

    public int getWidth() { return width; }
    public int getHeight() { return height; }

    public int area() { return width * height; }
}

// Subclass attempting to reuse Rectangle behaviour
public class Square extends Rectangle {
    public Square(int side) {
        super(side, side);
    }

    @Override
    public void setWidth(int w) {
        this.width = w;
        this.height = w;   // keeps square invariant
    }

    @Override
    public void setHeight(int h) {
        this.width = h;
        this.height = h;   // keeps square invariant
    }
}

Client code expecting a Rectangle:

public static void resizeAndCheck(Rectangle r) {
    r.setWidth(5);
    r.setHeight(10);
    // Expect area == 50 for a rectangle with width=5 height=10
    System.out.println("area = " + r.area());
}

If you pass a Square into resizeAndCheck, after setWidth(5) and setHeight(10) the Square will end up as width=10,height=10 (area=100). The client expected area==50. Violation of LSP.

After — LSP-focused refactor options

There are multiple valid ways to fix this depending on needs. Two common approaches:

Option A — Prefer composition or sibling types over inheritance

Make Shape interface and implement Rectangle and Square separately. Don't have Square extend Rectangle.

// Contract: Shape focuses on behavior clients actually need
public interface Shape {
    int area();
}

// Rectangle is its own class
public class Rectangle implements Shape {
    private final int width;
    private final int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }
    public int getWidth() { return width; }
    public int getHeight() { return height; }
    public int area() { return width * height; }
}

// Square is also its own class, but not a Rectangle
public class Square implements Shape {
    private final int side;

    public Square(int side) {
        this.side = side;
    }
    public int getSide() { return side; }
    public int area() { return side * side; }
}

Client code uses Shape:

public static void printArea(Shape s) {
    System.out.println("area = " + s.area());
}

Now substitutability is semantic: anything implementing Shape can be used. There's no swapped setter semantics because there are no setters.

When to use: When the conceptual contract differs or when mutability semantics differ between types.

Option B — Preserve behavioral contract in subtype (immutable or consistent setters)

If inheritance is needed, ensure subclass preserves all behavioral expectations: either make parent immutable or design setters so subclasses cannot break the contract.

Make Rectangle immutable or finalize setter behavior:

public class Rectangle {
    private int width;
    private int height;

    public Rectangle(int w, int h) { this.width = w; this.height = h; }

    public void setSize(int w, int h) {
        this.width = w;
        this.height = h;
    }

    public int getWidth() { return width; }
    public int getHeight() { return height; }
    public int area() { return width * height; }
}

// If Square must extend Rectangle, it must not change semantics of setSize
public class Square extends Rectangle {
    public Square(int side) { super(side, side); }

    @Override
    public void setSize(int w, int h) {
        if (w != h) throw new IllegalArgumentException("Square requires equal sides");
        super.setSize(w, h);
    }
}

This still can be problematic (now the subtype strengthens a precondition by rejecting w != h) — that is LSP-unsafe because callers of Rectangle.setSize may assume it never throws for valid inputs. So the safe approach is: do not strengthen preconditions. Better: avoid using inheritance here.

Conclusion: When the semantic contract differs, prefer interfaces/siblings or composition.

Concrete test that exposes the bug (unit test)

@Test
public void testResizeRectangle() {
    Rectangle r = new Rectangle(2, 3);
    r.setWidth(5);
    r.setHeight(10);
    assertEquals(50, r.area());  // passes
}

@Test
public void testResizeSquareAsRectangle() {
    Rectangle s = new Square(2);
    s.setWidth(5);
    s.setHeight(10);
    // This would fail if Square overrides setters to maintain square invariant.
    assertEquals(50, s.area()); // fails for the broken Square
}

If second test fails, your subtype is not substitutable.

Practical LSP guidelines for modern codebases

  • Prefer composition over inheritance when the subclass would need to change base behavior.
  • Design narrow interfaces. Clients should depend only on the minimum contract they need.
  • Immutability simplifies LSP. Immutable base types remove many invariants issues.
  • Be explicit about contracts in docs and types. Document preconditions, postconditions, and side effects.
  • Use final classes or sealed hierarchies when you need to control subclassing (Java final, sealed, or equivalent).
  • Avoid overriding methods that change behavior/side-effects (especially setters).
  • Prefer factories that return interfaces rather than concrete base classes if implementations may vary.
  • Property-based tests (QuickCheck-style) help detect substitution anomalies: run the same property tests for each implementation.

LSP-focused refactor recipe (step-by-step)

  1. Identify the contract: list the behavioral guarantees (pre/post/invariants) of the base type.
  2. Run base-type tests against each subtype. If they fail, you have an LSP problem.
  3. Decide semantics: Should subtype truly be a subtype (same semantics) or only share implementation?
  4. If semantics differ: refactor into separate implementations behind a common interface (composition).
  5. If semantics are same: enforce contract in subtype (but be cautious not to strengthen preconditions).
  6. Make invariants explicit: private fields, immutable where possible.
  7. Add tests that exercise substitution: run base-type test-suite against each implementation.
  8. Document contracts in code (javadoc/types, assertions).

Short lint-style checklist for code reviews (add to PR template)

  • Contract documented: Are preconditions/postconditions/invariants of the base type documented?
  • Behavioral substitution: Can an instance of the subtype be used anywhere the base type is used without changing observable behavior?
  • No strengthened preconditions: Does the subtype require stricter inputs than the base type?
  • No weakened postconditions: Does the subtype still guarantee what callers expect from the base type?
  • No hidden exceptions: Does the subtype introduce new unchecked exceptions or change exception semantics for existing methods?
  • Mutability match: Does the subtype preserve mutability/immutability expectations of the base?
  • Tests run polymorphically: Are base-type tests executed against the subtype implementation?
  • Prefer composition: If behavior differs, did the author consider composition or separate implementations instead of inheritance?
  • Final / sealed / restricted inheritance: Is inheritance intentionally allowed and controlled (use final, sealed, or restrict constructors)?
  • No state invariant leaks: Does subclass maintain base invariants (no internal state that violates base assumptions)?

Quick heuristics — red flags during review

  • Overriding setters and changing state semantics.
  • Throwing exceptions where base never did.
  • Overriding a method that callers assume is side-effect-free to add side-effects.
  • Public or protected fields that subclasses mutate in incompatible ways.
  • Subclass violates documented complexity/performance guarantees used by clients.

Closing example (summary)

  • Bad: Square extends Rectangle with different setter semantics — breaks LSP.
  • Good: Shape interface with Rectangle and Square as independent implementations — keeps substitution safe for the operations the client needs (e.g., area()).
  • Rule of thumb: If a subclass must change the behavior of base methods to preserve its own invariants, it's probably not a proper subtype — refactor.
