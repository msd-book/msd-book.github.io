Observer pattern — why, where, and how (with a ready-to-run Java refactor)

Short answer: use Observer when you have one subject (source of truth) and many interested consumers that should react to changes without the subject knowing concrete consumer types. It reduces coupling, improves testability, and fits UIs, event-systems, caching, reactive pipelines, or any publish/subscribe scenario.

Below I’ll show:

  • a before example that’s tightly coupled (easy to understand, hard to extend),
  • an after refactor implementing a clean Observer design (easy to extend, test, reuse),
  • notes on concurrency, pitfalls, and a short lint-style checklist for code reviews.

Before — tightly coupled code (anti-pattern)

This is a minimal, runnable Java example where StockTicker directly knows and calls a concrete StockDisplay. If you later add other displays, or non-UI consumers (logs, alerts), you must change StockTicker.

// File: BeforeObserver.java
public class BeforeObserver {
    public static void main(String[] args) {
        StockTicker ticker = new StockTicker();
        StockDisplay display = new StockDisplay();
        ticker.setDisplay(display);

        // Simulate price updates
        ticker.updatePrice("AAPL", 174.25);
        ticker.updatePrice("GOOG", 2830.50);
    }
}

class StockTicker {
    private StockDisplay display; // concrete dependency -> tight coupling

    public void setDisplay(StockDisplay display) {
        this.display = display;
    }

    public void updatePrice(String symbol, double price) {
        // compute, validate, etc...
        System.out.println("Ticker: price updated for " + symbol + " -> " + price);
        // directly call the concrete display
        if (display != null) {
            display.showPrice(symbol, price);
        }
    }
}

class StockDisplay {
    public void showPrice(String symbol, double price) {
        System.out.printf("Display: %s = %.2f%n", symbol, price);
    }
}

Problems with this design

  • StockTicker depends on concrete StockDisplay. Adding a logger, email alert, or another UI requires changing StockTicker.
  • Hard to unit-test StockTicker in isolation.
  • Only supports one display (or requires StockTicker to manage a collection of displays, but still tightly coupled to their types).

After — clean Observer refactor (recommended)

We define small interfaces Subject and Observer. StockMarket (subject) manages observers and notifies them. Observers implement StockObserver. This supports many observers, decouples publish/subscribe logic, and is easy to extend.

// File: AfterObserver.java
import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

public class AfterObserver {
    public static void main(String[] args) {
        StockMarket market = new StockMarket();
        market.registerObserver(new ConsoleDisplay("Console-1"));
        market.registerObserver(new LoggerObserver());
        market.registerObserver(new ThresholdAlert("AAPL", 200.0));

        market.updatePrice("AAPL", 174.25);
        market.updatePrice("GOOG", 2830.50);
        market.updatePrice("AAPL", 201.00); // triggers threshold alert
    }
}

/** Observer interface: implement this to react to updates */
interface StockObserver {
    void onPriceChange(String symbol, double price);
}

/** Subject interface */
interface StockSubject {
    void registerObserver(StockObserver o);
    void unregisterObserver(StockObserver o);
    void notifyObservers(String symbol, double price);
}

/** Concrete Subject: thread-safe-ish observer list using CopyOnWriteArrayList */
class StockMarket implements StockSubject {
    // CopyOnWriteArrayList is safe for concurrent reads and modifications and keeps iteration stable.
    private final List<StockObserver> observers = new CopyOnWriteArrayList<>();

    @Override
    public void registerObserver(StockObserver o) {
        if (o == null) throw new IllegalArgumentException("observer cannot be null");
        observers.add(o);
    }

    @Override
    public void unregisterObserver(StockObserver o) {
        observers.remove(o);
    }

    @Override
    public void notifyObservers(String symbol, double price) {
        for (StockObserver o : observers) {
            try {
                o.onPriceChange(symbol, price);
            } catch (Exception ex) {
                // Don't allow a bad observer to take down notification loop.
                System.err.println("Observer error: " + ex.getMessage());
                ex.printStackTrace();
            }
        }
    }

    // Domain method to update price
    public void updatePrice(String symbol, double price) {
        System.out.printf("StockMarket: %s -> %.2f%n", symbol, price);
        // Could apply filtering, batching, or debounce here before notify
        notifyObservers(symbol, price);
    }
}

/** Example observers */

class ConsoleDisplay implements StockObserver {
    private final String name;

    ConsoleDisplay(String name) { this.name = name; }

    @Override
    public void onPriceChange(String symbol, double price) {
        System.out.printf("[%s] Display: %s = %.2f%n", name, symbol, price);
    }
}

class LoggerObserver implements StockObserver {
    @Override
    public void onPriceChange(String symbol, double price) {
        // Pretend to log to a logging system
        System.out.println("Logger: recorded " + symbol + " at " + price);
    }
}

class ThresholdAlert implements StockObserver {
    private final String symbolToWatch;
    private final double threshold;

    ThresholdAlert(String symbolToWatch, double threshold) {
        this.symbolToWatch = symbolToWatch;
        this.threshold = threshold;
    }

    @Override
    public void onPriceChange(String symbol, double price) {
        if (symbol.equals(symbolToWatch) && price >= threshold) {
            System.out.printf("ALERT: %s crossed threshold %.2f (current: %.2f)%n",
                              symbol, threshold, price);
        }
    }
}

Why this is better

  • StockMarket doesn't need to know concrete observers.
  • Add/remove observers at runtime without changing subject code.
  • Observers can be reused in other contexts.
  • Easier to unit-test each observer and the subject independently.
  • Better separation of concerns and single responsibility.

Where to use Observer in modern systems

Use Observer or publish/subscribe in:

  • GUI: widgets reacting to model changes (MVVM/MVC).
  • Event-driven microservices: lightweight pub/sub between components (often via message brokers like Kafka/RabbitMQ).
  • Caching: cache invalidation when underlying data changes.
  • Metrics/logging/alerting: multiple consumers of events.
  • Reactive streams and stream processing (Rx, Reactor) — these are essentially a richer, composable evolution of Observer.

When the system needs more features (backpressure, composition, complex transformations) prefer reactive libraries (Reactor, RxJava) or messaging systems.

Implementation considerations & pitfalls

  • Thread-safety: use thread-safe observer collections (e.g., CopyOnWriteArrayList) or synchronize access if updates come from multiple threads. Avoid holding locks while calling observers.
  • Observer lifetime / memory leaks: ensure observers can be unregistered. Long-lived subjects holding references to short-lived observers (UI screens) cause leaks. Consider weak references if appropriate.
  • Fault isolation: wrap observer calls in try/catch so a failing observer doesn’t stop others.
  • Performance: notifying many observers synchronously can be slow. Consider async notification (executor) or batching if latency-sensitive.
  • Ordering: if order matters, document and design for it; CopyOnWriteArrayList preserves insertion order.
  • Backpressure & flow control: Observer pattern is basic; for heavy event rates, use reactive streams or message brokers that support backpressure.
  • Avoid business work on observer thread: observers should offload heavy processing to background threads if necessary.

Lint-style checklist for code reviews

Use this checklist when reviewing Observer-style code:

  1. Interface-based

    • ✔️ Subject and Observer are defined as interfaces (not concrete classes). This enables test doubles and decoupling.

  2. No concrete coupling

    • ✔️ Subject code does not reference concrete observer types or UI classes.

  3. Registration API

    • ✔️ There are register and unregister methods and sensible null/duplicate handling.

  4. Thread-safety

    • ✔️ Observer collection is thread-safe (or access is synchronized). Check for potential deadlocks and avoid holding locks while notifying.

  5. Exception handling

    • ✔️ Notifications to observers are wrapped so one bad observer can't break the whole notification loop.

  6. Memory leak prevention

    • ✔️ Observers are unregistered when no longer needed (e.g., UI tear-down). Prefer weak references only after explicit reasoning.

  7. No heavy work on notify path

    • ✔️ Notifications should be quick. Observers should offload heavy tasks (IO, DB calls) to their own executors.

  8. Backpressure / rate control

    • ✔️ For high-frequency events, design for batching, sampling, or use a reactive framework.

  9. Testing

    • ✔️ Unit tests: subject notifications are verified (mock observer), observers are tested in isolation, and lifecycle (register/unregister) is tested.

  10. Naming & docs

  • ✔️ Clear names and Javadocs: what events are sent, guarantee of ordering, threading model, and when observers are invoked.
  1. Use appropriate abstraction
  • ✔️ For complex event processing, prefer Rx/Reactor or messaging rather than raw Observer.

Quick decision guide

  • Use plain Observer when:

    • You need simple decoupling and a few observers.
    • Events are low-to-moderate rate and ordering is simple.

  • Use reactive streams / message broker when:

    • You need backpressure, complex transformations, durable delivery, or cross-process/eventual consistency.
