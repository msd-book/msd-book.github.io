Software Requirements Specification (SRS)

Project: Online Bookstore (Web Application) Version: 1.0 Date: 09 Dec 2025

1. Introduction

1.1 Purpose

This document specifies the functional and non-functional requirements for an Online Bookstore web application. It is intended for:

  • Developers implementing the system
  • Testers validating functionality
  • Product owners and stakeholders approving scope

1.2 Scope

The system allows users to:

  • Browse and search books
  • View book details
  • Purchase books online
  • Track orders

Out of scope:

  • Physical logistics and delivery optimization
  • Publisher onboarding workflows

1.3 Definitions and Acronyms

  • User: Any registered customer
  • Admin: Internal staff managing catalog and orders
  • SRS: Software Requirements Specification

2. Overall Description

2.1 Product Perspective

The Online Bookstore is a 3-tier web application:

  • Frontend (Web UI)
  • Backend API
  • Database

It integrates with:

  • Third-party payment gateway
  • Email notification service

Why this matters: This sets context, not implementation details.

2.2 User Classes and Characteristics

User Type Description Technical Skill
Customer Buys books online Low–Medium
Admin Manages catalog & orders Medium

2.3 Operating Environment

  • Web browsers: Chrome, Firefox, Edge (latest 2 versions)
  • Backend hosted on Linux servers
  • Database: Relational (e.g., PostgreSQL)

2.4 Constraints

  • Must comply with Indian IT Act for data protection
  • Payment processing via PCI-compliant gateway
  • Initial release must support up to 10,000 registered users

3. Functional Requirements

Rule of thumb: Functional requirements describe what the system does, not how.

FR-1: User Registration

Description: The system shall allow users to register using email and password.

Inputs:

  • Email
  • Password

Outputs:

  • Confirmation email
  • User account created

Acceptance Criteria:

  • Email must be unique
  • Password must be at least 8 characters

FR-2: Book Search

Description: The system shall allow users to search books by title, author, or ISBN.

Priority: High

FR-3: Place Order

Description: The system shall allow a logged-in user to place an order for selected books.

Preconditions:

  • User is authenticated
  • Book is in stock

Postconditions:

  • Order is created
  • Payment is processed
  • Confirmation email is sent

FR-4: Admin Manages Catalog

Description: The system shall allow admins to add, update, or remove books from the catalog.

4. Non-Functional Requirements

This is where many engineers under-specify. These often matter more than features.

4.1 Performance

  • Search results shall be returned within 2 seconds for 95% of requests.
  • System shall support 100 concurrent users without degradation.

4.2 Security

  • All passwords must be stored using salted hashing.
  • All communication must use HTTPS.
  • Users shall not access admin APIs.

4.3 Availability

  • System uptime shall be 99.5% on a monthly basis.

4.4 Usability

  • A first-time user should be able to place an order in under 5 minutes.
  • Error messages must be clear and actionable.

4.5 Maintainability

  • Codebase shall follow modular architecture.
  • APIs shall be versioned.

5. External Interface Requirements

5.1 User Interface

  • Web UI with responsive design
  • Supports desktop and mobile screens

5.2 API Interfaces

  • RESTful APIs over HTTPS
  • JSON request/response format

5.3 Third-Party Interfaces

  • Payment Gateway API
  • Email Service API

6. Assumptions and Dependencies

  • Internet connectivity is available to users
  • Payment gateway uptime is outside system control

7. Acceptance Criteria Summary

The system is considered acceptable when:

  • All functional requirements are implemented
  • Non-functional metrics are met in production
  • Security and performance tests pass

What You Should Learn From This Example

  1. Clarity beats cleverness

    • Plain language is preferred over jargon.

  2. Separation of concerns

    • Requirements ≠ design ≠ implementation.

  3. Testability matters

    • Good SRS statements can be turned into test cases directly.

  4. Non-functional requirements are not optional

    • Performance, security, and usability must be explicit.

  5. Scope control

    • Out-of-scope sections prevent future confusion.

4. What You Should Notice

  1. One requirement → many test cases

    • Happy path + edge cases + failures.

  2. Non-functional requirements are testable

    • They need tools, not UI clicks.

  3. Clear SRS enables easy testing

    • Ambiguous requirements produce weak tests.

  4. Traceability is non-negotiable

    • Every requirement must be verifiable.

5. Simple Rule to Remember

If you can’t write a test case for it, it doesn’t belong in the SRS.

High-Level Design (HLD)

Project: Online Bookstore Version: 1.0

1. Purpose of This Document

This document describes the overall system architecture and major components of the Online Bookstore system. It explains how the system is structured, not how each class or function is written.

Mental model: HLD answers “What are the big building blocks and how do they talk?”

2. System Overview

2.1 Architectural Style

  • Three-Tier Architecture

    • Presentation Layer
    • Application (Service) Layer
    • Data Layer

2.2 Technology Stack (Indicative)

  • Frontend: Web SPA (React / Angular)

  • Backend: REST API (ASP.NET Core / Java Spring)

  • Database: PostgreSQL

  • External Services:

    • Payment Gateway
    • Email Service

(Exact tech can change without violating HLD)

4. Major Components

4.1 Presentation Layer

Responsibilities

  • Render UI screens
  • Capture user input
  • Call backend APIs

Key Modules

  • Authentication UI
  • Book Catalog UI
  • Shopping Cart UI
  • Admin Dashboard

4.2 Application Layer

Responsibilities

  • Business logic
  • Validation
  • Authorization
  • Orchestration of workflows

Key Services

  • User Service
  • Catalog Service
  • Order Service
  • Payment Service
  • Admin Service

4.3 Data Layer

Responsibilities

  • Persistent storage
  • Data integrity

Key Entities

  • Users
  • Books

  1. User clicks “Checkout”

  2. Frontend sends request to Order API

  3. Order Service:

    • Validates user
    • Checks stock

  4. Order stored in DB

  5. Confirmation email sent

Quality Attribute Design Choice

Low-Level Design (LLD)

Project: Online Bookstore Version: 1.0

1. Purpose of This Document

This document provides detailed design of individual components, classes, APIs, and database schema.

Mental model: LLD answers “How exactly do we build each part?”

2. API Design

2.1 User Registration API

Endpoint

POST /api/users/register

Request

{
  "email": "user@example.com",
  "password": "password123"
}

Response

{
  "userId": "uuid",
  "status": "REGISTERED"
}

Validations

  • Email uniqueness
  • Password length ≥ 8

2.2 Book Search API

GET /api/books?query={text}

Behavior

  • Search by title, author, ISBN
  • Pagination supported

2.3 Place Order API

POST /api/orders

Request

{
  "items": [
    { "bookId": "uuid", "quantity": 1 }
  ]
}

Steps

  1. Validate user session
  2. Validate stock
  3. Create order
  4. Initiate payment

3. Database Design

Column Type
