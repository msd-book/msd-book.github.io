Adapter pattern — why, where, and how (practical C# example)

Short version: use the Adapter when you need to make an existing class or third-party API look like the interface your code expects — without changing either side. It’s about translation: convert calls, data shapes, and semantics so the rest of your app remains clean and decoupled.

Common situations:

  • Integrating a third-party library whose API doesn’t match your interfaces.
  • Replacing a legacy implementation gradually while keeping the rest of the system unchanged.
  • Making multiple different implementations appear under a single interface (swap at runtime, test, fallback).

Two styles:

  • Object adapter (composition) — most common in C#. The adapter holds an instance of the incompatible class and translates calls.
  • Class adapter (inheritance) — less common in C# because of single inheritance constraints; possible via multiple interfaces or subclassing, but tends to be brittle.

Before / After — real C# example

Scenario: your app needs a shipping carrier abstraction ICarrier.Ship(Order). The new carrier SDK exposes a different API (ThirdPartyCarrier.SendPackage(PackageInfo)). The app currently calls the third-party directly (tight coupling). We'll refactor with an adapter so the app depends only on ICarrier.

All code is a minimal .NET 6+ console example — copy, paste, run.

Before — tight coupling (bad)

// ProgramBefore.cs
using System;

namespace AdapterBefore
{
    // Domain model
    public class Order
    {
        public string OrderId { get; init; }
        public string Address { get; init; }
        public double WeightKg { get; init; }
    }

    // Third-party carrier (incompatible)
    public class ThirdPartyCarrier
    {
        public ThirdPartyResponse SendPackage(PackageInfo package)
        {
            // pretend this calls remote API
            Console.WriteLine($"ThirdPartyCarrier: sending package {package.Reference} to {package.Destination}");
            return new ThirdPartyResponse { Status = "OK", TrackingNumber = "TP-" + package.Reference };
        }
    }

    public class PackageInfo
    {
        public string Reference { get; init; }
        public string Destination { get; init; }
        public double MassInKg { get; init; }
    }

    public class ThirdPartyResponse
    {
        public string Status { get; init; }
        public string TrackingNumber { get; init; }
    }

    // Service that uses third party directly -> NOT testable, hard to swap
    public class ShippingService
    {
        private readonly ThirdPartyCarrier _carrier = new ThirdPartyCarrier();

        public string ShipOrder(Order order)
        {
            var pkg = new PackageInfo
            {
                Reference = order.OrderId,
                Destination = order.Address,
                MassInKg = order.WeightKg
            };

            var response = _carrier.SendPackage(pkg);

            if (response.Status != "OK")
                throw new Exception("Shipping failed");

            return response.TrackingNumber;
        }
    }

    class Program
    {
        static void Main()
        {
            var service = new ShippingService();
            var tracking = service.ShipOrder(new Order { OrderId = "1001", Address = "1 Main St", WeightKg = 2.3 });
            Console.WriteLine($"Tracking: {tracking}");
        }
    }
}

Problems:

  • ShippingService is tightly coupled to ThirdPartyCarrier.
  • Hard to unit test ShippingService (can't mock carrier easily).
  • Replacing carrier requires editing ShippingService.

After — adapter applied (good)

We introduce ICarrier (the app interface), an adapter ThirdPartyCarrierAdapter : ICarrier that translates Order → PackageInfo and ThirdPartyResponse → ShipmentResult. ShippingService depends on ICarrier (injected) — easily testable and swappable.

// ProgramAfter.cs
using System;

namespace AdapterAfter
{
    // Domain model
    public class Order
    {
        public string OrderId { get; init; }
        public string Address { get; init; }
        public double WeightKg { get; init; }
    }

    public class ShipmentResult
    {
        public bool Success { get; init; }
        public string TrackingNumber { get; init; }
        public string Error { get; init; }
    }

    // The interface the rest of the app expects
    public interface ICarrier
    {
        ShipmentResult Ship(Order order);
    }

    // Third-party carrier (unchangeable)
    public class ThirdPartyCarrier
    {
        public ThirdPartyResponse SendPackage(PackageInfo package)
        {
            // pretend this calls remote API
            Console.WriteLine($"ThirdPartyCarrier: sending package {package.Reference} to {package.Destination}");
            return new ThirdPartyResponse { Status = "OK", TrackingNumber = "TP-" + package.Reference };
        }
    }

    public class PackageInfo
    {
        public string Reference { get; init; }
        public string Destination { get; init; }
        public double MassInKg { get; init; }
    }

    public class ThirdPartyResponse
    {
        public string Status { get; init; }
        public string TrackingNumber { get; init; }
    }

    // The Adapter: converts between our ICarrier and ThirdPartyCarrier
    public class ThirdPartyCarrierAdapter : ICarrier
    {
        private readonly ThirdPartyCarrier _thirdParty;

        public ThirdPartyCarrierAdapter(ThirdPartyCarrier thirdParty)
        {
            _thirdParty = thirdParty ?? throw new ArgumentNullException(nameof(thirdParty));
        }

        public ShipmentResult Ship(Order order)
        {
            if (order == null) throw new ArgumentNullException(nameof(order));

            // mapping domain Order -> third-party PackageInfo
            var package = new PackageInfo
            {
                Reference = order.OrderId,
                Destination = order.Address,
                MassInKg = order.WeightKg
            };

            try
            {
                var resp = _thirdParty.SendPackage(package);

                // map third-party response -> domain ShipmentResult
                return new ShipmentResult
                {
                    Success = resp.Status == "OK",
                    TrackingNumber = resp.TrackingNumber,
                    Error = resp.Status == "OK" ? null : $"Carrier error: {resp.Status}"
                };
            }
            catch (Exception ex)
            {
                // adapter shields the rest of app from third-party exceptions
                return new ShipmentResult
                {
                    Success = false,
                    Error = $"Carrier exception: {ex.Message}"
                };
            }
        }
    }

    // ShippingService now depends on ICarrier -> DIP, easy to test & swap
    public class ShippingService
    {
        private readonly ICarrier _carrier;

        public ShippingService(ICarrier carrier)
        {
            _carrier = carrier ?? throw new ArgumentNullException(nameof(carrier));
        }

        public string ShipOrder(Order order)
        {
            var result = _carrier.Ship(order);

            if (!result.Success)
            {
                throw new InvalidOperationException($"Shipping failed: {result.Error}");
            }

            return result.TrackingNumber;
        }
    }

    // Example usage
    class Program
    {
        static void Main()
        {
            var thirdParty = new ThirdPartyCarrier();
            ICarrier adapter = new ThirdPartyCarrierAdapter(thirdParty);
            var service = new ShippingService(adapter);

            var order = new Order { OrderId = "1001", Address = "1 Main St", WeightKg = 2.3 };
            var tracking = service.ShipOrder(order);
            Console.WriteLine($"Tracking: {tracking}");

            // Replace adapter at runtime if needed (e.g., for testing)
            ICarrier fakeCarrier = new InMemoryCarrier();
            var testService = new ShippingService(fakeCarrier);
            Console.WriteLine("Test tracking: " + testService.ShipOrder(order));
        }
    }

    // A simple in-memory carrier for tests or local fallback
    public class InMemoryCarrier : ICarrier
    {
        public ShipmentResult Ship(Order order) =>
            new ShipmentResult { Success = true, TrackingNumber = "INMEM-" + order.OrderId };
    }
}

Why this refactor is better:

  • ShippingService depends on the abstraction ICarrier (Dependency Inversion).
  • Adapter isolates mapping and error handling from business logic.
  • Swapping carriers or stubbing in tests is trivial.
  • You can gradually add adapters for other carriers without changing ShippingService.

Practical guidance — how to implement an adapter well

  1. Keep adapter responsibility limited

    • Adapter should translate calls and data only — don’t put business logic there.

  2. Encapsulate mapping logic

    • Centralize mapping code inside the adapter to avoid scattering conversions.

  3. Normalize error handling & exceptions

    • Adapter should convert third-party errors into your domain error/result types.

  4. Make adapters small and testable

    • Unit test mapping and error-paths in the adapter; integration tests verify real carrier behavior.

  5. Use dependency injection

    • Inject the adapter as the ICarrier implementation; makes swapping trivial.

Lint-style checklist for code reviews

  • Abstraction exists: Is there a clear interface/protocol the rest of the app uses (e.g., ICarrier)?
  • Adapter encapsulates translation: Is mapping of data models and statuses confined to the adapter (not in callers)?
  • No business logic leaked: Adapter contains only translation and error normalization — no domain rules.
  • Exceptions translated: Third-party exceptions do not leak; adapter normalizes errors to domain types.
  • Thin adapter: Adapter is small and focused; complex mapping factored into helper methods or mappers.
  • DI-ready: The adapter (or interface) can be injected; no direct new ThirdPartyCarrier() inside business services.
  • Test coverage: Adapter has unit tests for success + failure mapping; service using adapter has tests with fakes/mocks.
  • No direct imports of third-party types across boundary: Consuming code should not depend on third-party SDK types.
  • Performance check: Adapter does not introduce heavy synchronous blocking or large allocations unintentionally.
  • Logging and metrics: Important adapter events (mapping failures, timeouts) are logged or instrumented appropriately.

Common pitfalls & how to avoid them

  1. Over-adapting / multiple responsibilities

    • Pitfall: Adapter becomes a place to hide significant business logic.
    • Avoid: Keep business rules out; adapter = adapter (translate and normalize).

  2. Leaking third-party types through your API

    • Pitfall: Domain code starts referencing PackageInfo or ThirdPartyResponse.
    • Avoid: Map to domain DTOs and return domain results.

  3. Duplicate mapping code

    • Pitfall: Several adapters or services do the same conversions.
    • Avoid: Share mapping helpers or a small mapper class.

  4. Swapping performance assumptions

    • Pitfall: New adapter introduces blocking calls, slow serialization, or many network round trips.
    • Avoid: Profile, make adapters async if necessary, and batch work where appropriate.

  5. Ignoring failure modes

    • Pitfall: Adapter returns raw exceptions or nulls on errors; callers crash.
    • Avoid: Normalize to ShipmentResult/error objects and add retries/timeouts inside the adapter if appropriate.

  6. Overuse of Adapter when a Facade/Bridge is better

    • Pitfall: Using adapter to hide a whole subsystem or to add new functionality.
    • Avoid: If you need to provide a simpler API over a set of complex classes, consider Facade. If you need to vary implementations orthogonally, consider Bridge.

  7. Too much translation in hot paths

    • Pitfall: Heavy serialization/deserialization or mapping on critical paths causing latency.
    • Avoid: Optimize mapping (avoid reflection), use pooled objects, or move expensive transforms off the hot path.

Quick decision guide: Adapter vs other patterns

  • Adapter — translate one interface to another (use this when the interface mismatch is the issue).
  • Facade — provide a simpler interface to a complex subsystem (use when you need to simplify).
  • Decorator — extend behavior dynamically (use when you want to add responsibilities).
  • Bridge — separate abstraction from implementation so both can vary independently (use when you expect many implementations and abstractions to evolve).

Final checklist for adoption

  • Identify the boundary interface (ICarrier) and ensure domain code depends only on it.
  • Create adapters that implement the domain interface and encapsulate all translation/error handling.
  • Add unit tests for adapters (mapping success/failure) and services (use fakes).
  • Wire via DI and keep original third-party SDK usage confined to adapter(s).
